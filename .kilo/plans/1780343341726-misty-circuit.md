## 1. Visión General

### 1.1 Producto

Vantare Overlays es una aplicación de escritorio profesional para simracing que proporciona
overlays de telemetría en tiempo real para streaming (OBS Studio) y uso multi-monitor.
Compitiendo directamente con RaceLabs, se diferencia por su soporte multi-sim (iRacing + LMU + AC),
theme system completo, stream alerts nativos y modelo freemium.

### 1.2 MVP (v1.0)

| Componente | Alcance |
|---|---|
| **Sims** | iRacing, Le Mans Ultimate, Assetto Corsa |
| **Overlays** | Standings, Relative, Delta Bar, Stream Alerts |
| **Dashboard** | Tab dentro de Electron (Hub con 5 páginas) |
| **Temas** | 3 themes built-in + custom (CSS variables + Tailwind v4) |
| **Auth** | Supabase Auth + PostgreSQL + Feature Gating |
| **Distribución** | `.exe` via Electron Builder + Auto-update |
| **Streaming** | HTTP Server (localhost:3200) + SSE + OBS Browser Sources |

### 1.3 Métricas de Éxito

| Métrica | Target |
|---|---|
| Renderizado de overlay | < 16ms (60 FPS) |
| Memoria renderer | < 150MB |
| Arranque app | < 3 segundos |
| Cobertura E2E | > 70% |
| Tasa de errores en beta | < 1% |

### 2.2 Flujo de Datos (High-Level)

```
SIM (Shared Memory) → N-API C++ Adapter → Normalizer → UnifiedTelemetry
    → SimManager → IPC (contextBridge) → Zustand Store → React Overlay (render)
    → HTTP Server (SSE) → OBS Browser Source
```

## 6. Telemetry Pipeline

### 6.1 Pipeline Steps

```
Step 1: Sim writes to shared memory (iRacing/LMU) or sends UDP (AC)
Step 2: N-API C++ addon reads via mmap (zero-copy)
Step 3: Adapter calls callback with raw sim data
Step 4: Normalizer maps sim-specific fields → UnifiedTelemetry
Step 5: SimManager receives UnifiedTelemetry
Step 6: SimManager broadcasts via IPC (ipcMain → ipcRenderer)
Step 7: Preload script receives and forwards to renderer
Step 8: Zustand telemetryStore.setTelemetry(data)
Step 9: React overlays re-render via useTelemetry() selector
```

### 6.2 Unified Telemetry Type

```typescript
export interface Telemetry {
  sim: 'iracing' | 'lmu' | 'ac';
  timestamp: number;
  isConnected: boolean;
  player: PlayerData;      // speed, rpm, gear, position, fuel
  engine: EngineData;      // rpm, maxRpm, fuel levels, temps
  tyres: TyreData;         // fl, fr, rl, rr (temp, pressure, wear)
  lap: LapData;            // current, best, last, sector, delta
  session: SessionData;    // type, state, time, flags
  vehicles: VehicleData[]; // all drivers with position, gap, laps
  track: TrackData;        // name, length, sectors
  inputs: InputData;       // throttle, brake, clutch, steering
  weather: WeatherData;    // air, track temp, humidity, wind
}
```

### 6.3 Update Rate

| Rate | Constraint |
|---|---|
| 16 Hz (60ms) | Target update rate for overlays |
| 60 Hz | iRacing SDK native rate |
| 16 Hz | LMU shared memory refresh |
| 60 Hz | AC UDP packet rate |
| Zustand selector | Only re-render on changed fields |
| React.memo | All overlay components |

### 6.4 Mock System

Every adapter has a `mock.ts` that generates realistic telemetry data:
- Configurable scenarios: practice, qualifying, race
- Realistic driver names, lap times, gaps
- Simulates session progression
- Enables development without any sim installed
- Used by Storybook and unit tests

## 7. Multi-Sim Adapter Layer

### 7.1 SimAdapter Interface (already defined in `packages/sim-core/src/adapters/base.ts`)

```typescript
export interface SimAdapter {
  readonly name: SimType;
  readonly displayName: string;
  isAvailable(): boolean;
  connect(): Promise<void>;
  disconnect(): void;
  onTelemetry(callback: (data: RawTelemetry) => void): () => void;
  onSessionData(callback: (data: RawSession) => void): () => void;
  onConnectionState(callback: (state: ConnectionState) => void): () => void;
  getSimInfo(): SimInfo;
  destroy(): void;
}
```

### 7.2 SimManager

- Polls every 1000ms to check which sim is running
- Priority: iRacing > LMU > AC
- Auto-connects to detected sim
- Disconnects gracefully when sim closes
- Broadcasts sim state via IPC
- Retries on failure with exponential backoff

### 7.3 iRacing Adapter

- Reads shared memory via C++ N-API addon
- Uses `irsdk-node` or custom N-API binding
- Variable names: `Speed`, `RPM`, `Throttle`, `Brake`, `Clutch`, `Gear`, `FuelLevel`, `Lap`, `LapTime`, `LastLapTime`, `BestLapTime`, etc.
- Session data: `SessionType`, `SessionState`, `SessionTimeRemain`, `CarsInSession`
- Mock available for macOS/development

### 7.4 LMU Adapter

- Reads shared memory via C++ N-API addon (LMU shared memory interface v1.2+)
- Or via Python sidecar + FFI (alternative approach)
- Mock available for development

### 7.5 AC Adapter

- Reads UDP packets (port 9996 by default)
- Binary protocol defined by AC telemetry
- Mock available for development

### 7.6 Normalizer

One normalizer per sim, mapping sim-specific fields to UnifiedTelemetry:
- `normalizeIRacing(raw: IRacingTelemetry): Telemetry`
- `normalizeLMU(raw: LMUTelemetry): Telemetry`
- `normalizeAC(raw: ACTelemetry): Telemetry`
- Unit conversions: m/s → km/h, mi/h → km/h, F → C, PSI → kPa
- Default values for missing fields
- Vehicle data normalization (position, gaps, colors)

## 11. HTTP Server + OBS Integration

### 11.1 Architecture

```
HttpServer (apps/desktop/src/main/server/http-server.ts)
  ├── GET /events → SSE stream (text/event-stream)
  │     └── Clients connect, receive telemetry in real-time
  │     └── CORS headers for local network access
  │
  ├── GET /overlays/:id → HTML page
  │     └── Inline HTML with <style> and <script>
  │     └── EventSource connects to /events
  │     └── Background: transparent
  │
  └── broadcastTelemetry(data: Telemetry)
        └── Writes to all connected SSE clients
```

### 11.2 OBS Integration

User workflow:
1. Open Vantare app
2. Go to Hub → enable HTTP server
3. Copy overlay URL: `http://localhost:3200/overlays/standings`
4. In OBS: Add Browser Source → paste URL
5. Set width/height, check "Transparent background"
6. Done — overlay appears on stream

### 11.3 Implementation

Already stubbed in `apps/desktop/src/main/server/http-server.ts`:
- `start()` — Creates http.Server, listens on port 3200
- `stop()` — Closes server and client connections
- `broadcastTelemetry()` — SSE write to all clients
- `renderOverlayPage()` — Generates HTML for each overlay

To implement fully:
- Serve actual Vite-built overlay bundles
- Handle port conflicts (auto fallback)
- Network access toggle (0.0.0.0 vs 127.0.0.1)
- Connection monitoring

### 16.2 F-018: Overlay Configuration System

**Origen**: `docs/OVERLAY-DEV-GUIDE.md` Section 6
**Objetivo**: Sistema tipado de configuración por overlay con Zod schemas y UI auto-generada.

**Comportamiento**:
- Cada overlay exporta un Zod schema de configuración
- El Hub auto-genera la UI de settings a partir del schema
- Settings se persisten via IPC → electron-store
- Valores por defecto definidos en el schema
- Settings changes se propagan via IPC push al overlay

**Patrón de implementación**:
```typescript
// standings.config.ts
const standingsConfigSchema = z.object({
  maxDrivers: z.number().min(3).max(50).default(20),
  showIRating: z.boolean().default(true),
  nameFormat: z.enum(['full', 'first_initial', 'last_initial']).default('full'),
  compactMode: z.enum(['off', 'compact', 'ultra']).default('off'),
  columnOrder: z.array(z.string()).default(['pos', 'car', 'driver', 'gap', 'irating']),
  highlightPlayer: z.boolean().default(true),
  showFlags: z.boolean().default(true),
});
```

**Depende de**: F-003 (Overlay System)
**Sprint**: Sprint 4 (con Hub Settings)
